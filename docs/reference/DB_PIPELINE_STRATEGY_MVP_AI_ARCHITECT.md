# DB Pipeline Strategy - MVP + AI Architect 방향

## 목표

이 프로젝트의 목표는 단순한 데이터 수집이 아니다.

- AI 기반 데이터 파이프라인 구축 능력 향상
- DB -> JSON -> MVP 연결 흐름 완성
- 궁극적으로 AI/데이터 아키텍트 역량 확보

동시에 아래 목표도 함께 가진다.

- 치킨 창업 시뮬레이터 MVP를 실제로 작동시키기
- 사용자 입장에서 의미 있는 정보 제공하기

## 현재 전략

이 프로젝트는 두 축을 동시에 진행한다.

## 1. DB 파이프라인

핵심 역량을 기르는 축이다.

- 데이터 수집
- 파싱
- 구조화
- JSON export
- 자동화 기반 구축

## 2. MVP

검증용 화면을 만드는 축이다.

- UX/UI 개선
- 사용자 판단 가능 구조
- 빠른 실험
- 필요하면 하드코딩도 허용

## 핵심 원칙

## 1. 완벽한 DB를 만들려고 하지 않는다

하지 않을 것:

- 모든 데이터를 한 번에 수집
- 모든 출처를 동시에 연결
- 처음부터 완성형 DB 설계에 매달리기

대신 작은 단위로 반복한다.

```text
작게 만들기
-> 확인하기
-> 고치기
-> 다음 데이터 하나 추가하기
```

## 2. 출처별로 한 번에 하나씩 확장한다

현재 기준 출처:

```text
공정위 데이터
```

다음 단계 후보:

```text
뉴스
통계청
```

하지만 한 번에 둘 다 붙이지 않는다.

예시:

```text
뉴스 -> marketTrend 항목 1개만 추가
통계청 -> population 항목 1개만 추가
```

이렇게 해야 오류가 났을 때 원인을 찾기 쉽다.

## 3. 데이터 상태를 같이 관리한다

각 값은 값 자체만 저장하지 않고, 그 값이 얼마나 믿을 수 있는지도 함께 관리한다.

기본 상태:

```text
official = 공식 데이터
estimate = 추정 데이터
unknown = 빈값 또는 알 수 없음
```

예시:

```json
{
  "averageSalesKrw": {
    "value": 450859000,
    "status": "official",
    "source": "공정위 정보공개서"
  },
  "marketTrend": {
    "value": "배달 수요는 유지되지만 원가 부담이 큼",
    "status": "estimate",
    "source": "뉴스 요약"
  },
  "population": {
    "value": null,
    "status": "unknown",
    "source": null
  }
}
```

## 현재 파이프라인 위치

현재 프로젝트는 아래 흐름 중 MVP 연결 직전까지 왔다.

```text
source data
-> parsing/filtering
-> LLM interpretation
-> database writing
-> JSON export
-> MVP connection
```

현재 완료된 부분:

```text
공정위 데이터 수집
필터링
Gemini 분석
Supabase 저장
MVP용 JSON 생성
```

다음 핵심 작업:

```text
MVP용 JSON 한글 깨짐 확인/수정
MVP 화면에서 JSON 읽기
4개 브랜드 카드 표시 확인
```

## 앞으로의 확장 순서

추천 순서:

1. 공정위 기반 4개 브랜드 MVP 연결 확인
2. JSON 한글/문구 품질 수정
3. 20개 브랜드로 확장
4. 뉴스 데이터에서 `marketTrend` 1개 항목 추가
5. 통계청 데이터에서 `population` 1개 항목 추가
6. 각 항목에 `official`, `estimate`, `unknown` 상태 붙이기

## 판단 기준

이 프로젝트에서 좋은 선택은 복잡한 선택이 아니라, 다음 조건을 만족하는 선택이다.

- 다음 단계로 진행 가능하다
- 오류가 보인다
- 다시 실행할 수 있다
- 사용자가 MVP에서 차이를 느낄 수 있다
- 내가 DB 파이프라인 개념을 하나 더 배운다

## 결론

이 프로젝트의 방향은 다음과 같다.

```text
완벽한 데이터베이스를 한 번에 만들기
```

가 아니라,

```text
작은 데이터 흐름을 끝까지 연결하고,
그 흐름을 출처별로 하나씩 확장하기
```

이다.

현재 가장 중요한 다음 단계는 공정위 기반 데이터를 MVP에서 정상적으로 보여주는 것이다.
